Convención de la columna "AOT C"

Cada opcode incluye su traducción a C para el generador AOT (H3). Las categorías visualmente marcadas:

  • inline C directo, 1-5 líneas. El AOT lo emite literal.
  • helper Llamada a una función vm_op_* del runtime. Usado cuando la lógica es compleja (GC, dispatch virtual, excepciones, range checks).
  • ctrl Control flow — el AOT lo traduce a estructuras C nativas (goto, return, llamada directa).
  • n/a No aplica al cuerpo de funciones AOT (HALT, THREAD_EXIT).

Variables en scope de la función AOT

uint8_t* mem;          // memory[] de la VM
uint32_t sp, bp, cs;   // registros BP del frame actual
bpvm_t*  vm;           // contexto VM (para helpers / GC / sink)

Helpers asumidos

static inline int32_t  rd32(uint8_t* p);                  // read i32 BE
static inline void     wr32(uint8_t* p, int32_t v);       // write i32 BE
static inline int16_t  rd16s(uint8_t* p);                 // read i16 BE sign-ext
static inline uint16_t rd16u(uint8_t* p);                 // read u16 BE
static inline float    i2f(int32_t b);                    // bitcast i32→f32
static inline int32_t  f2i(float f);                      // bitcast f32→i32

Los vm_op_* son funciones del runtime VM-C extraídas del switch del intérprete; cada una replica un bloque case OP_X: de interp.c. El refactor a static inline compartido entre intérprete y AOT es el primer paso de #157.

Convenciones de notación (bytecode)

i32, u32, i16, u16, i8, u8, f32
Enteros y float big-endian.
soff
Signed offset al bp (positivo = arg/var arriba; negativo = saved regs / args debajo).
rel
Relativo al instruction address (byte del opcode, NO al pc post-operando). Difiere del estilo JVM.

instruction_addr = pc - 1  # ya consumimos el byte de opcode
rel = read_immediate(pc)
pc = instruction_addr + rel

Aplica a JUMP, JUMP_IF_FALSE y sus variantes compactas.
csOff
Relativo al cs (= codeStart) del módulo actual.
Stack effect
Notación ( a b -- c ) = pop b, pop a, push c. La cima a la derecha.

Tras ejecutar un opcode, pc apunta al siguiente, salvo control flow.

ABI de frontera AOT ↔ BP (H3)

Cuando una función está marcada native y compilada a AOT, cohabita con código BP interpretado. Las llamadas entre los dos mundos cruzan una frontera de ABI: la convención BP usa el stack BP (sp/bp/cs/pc por memoria); la AOT usa la C ABI nativa (args en registros, return en r0). El runtime resuelve la llamada en tiempo de ejecución según el tipo de cada función.

Los 4 patrones de llamada

          caller             callee
        ┌──────────┐    ┌──────────┐
①  BP → │ OP_CALL  │───▶│ bytecode │   convención BP estándar (sin cambios)
        └──────────┘    └──────────┘

        ┌──────────┐    ┌──────────┐
②  BP → │ OP_CALL  │───▶│  thunk   │──▶ aot_fn(vm, a, b)
        └──────────┘    └──────────┘   thunk lee args del stack BP,
                                        llama C, escribe ret al stack BP.

        ┌──────────┐    ┌──────────┐
③ AOT → │ aot_a    │───▶│ aot_b    │   C call directa.
        └──────────┘    └──────────┘   Compilador puede inlinear y regalloc.

        ┌──────────┐    ┌──────────┐
④ AOT → │vm_call_bp│───▶│ bytecode │   helper materializa frame BP,
        └──────────┘    └──────────┘   ejecuta interp, recoge ret.

Tabla de dispatch del runtime

Por cada función importable, el módulo (al cargar) rellena una entry indicando si es BP o AOT:

typedef struct {
    enum { KIND_BP, KIND_AOT } kind;
    union {
        uint32_t bp_pc;                  // bytecode entry (CS-relative)
        struct {
            void* aot_fn;                // aot_Module_func directo
            void  (*thunk)(bpvm_t*, uint32_t* sp_p, uint32_t* bp_p);
        } aot;
    };
} bpvm_func_entry_t;

OP_CALL en el intérprete (modificación)

case OP_CALL: {
    int32_t rel = rd32(mem+pc); pc += 4;
    bpvm_func_entry_t* fe = bpvm_resolve_call(vm, cs, rel);
    if (fe->kind == KIND_AOT) {
        // BP→AOT (patrón ②) — thunk se encarga.
        fe->aot.thunk(vm, &sp, &bp);
        // NO se pushean saved pc/bp/cs. El AOT 'flota' sobre el stack
        // BP del caller sin tocarlo. pc del caller continúa intacto.
    } else {
        // BP→BP (patrón ①) — convención estándar.
        wr32(mem+sp, pc); sp += 4;
        wr32(mem+sp, bp); sp += 4;
        wr32(mem+sp, cs); sp += 4;
        bp = sp;
        pc = cs + fe->bp_pc;
    }
    break;
}

Generación del thunk (lado AOT emitter)

Por cada función native, el compilador emite dos cosas:

// 1) La función AOT con C-ABI nativa.
//    Locales BP → C locals (compiler regalloc). Recursión = C call directa.
int32_t aot_Bench_fib(bpvm_t* vm, int32_t n) {
    if (n < 2) return n;
    return aot_Bench_fib(vm, n - 1) + aot_Bench_fib(vm, n - 2);
}

// 2) El thunk para el patrón ② (BP→AOT).
//    Lee args del top del BP stack, llama AOT, escribe ret.
//    El interp llama esto en vez de hacer CALL/ENTER/RET para esta función.
void thunk_Bench_fib(bpvm_t* vm, uint32_t* sp_p, uint32_t* bp_p) {
    uint8_t* mem = vm->memory;
    uint32_t sp = *sp_p;
    int32_t n = rd32(mem + sp - 4);     // pop arg (1 i32)
    sp -= 4;
    int32_t r = aot_Bench_fib(vm, n);   // C call al AOT
    wr32(mem + sp, r); sp += 4;         // push result
    *sp_p = sp;
    // bp no cambia — el thunk no creó frame en el stack BP.
}

Acceso a sp/bp/cs/pc desde dentro del AOT

pc
Nunca. El AOT es C — el control flow es C (goto, return, function call).
cs
Sólo si lee globals. Hardcoded en el .c generado (cs del módulo conocido en codegen), o param.
bp
Nunca con AOT "lifted" (locales BP = locales C, no via memoria). Sí con AOT "transcribed" pero pierdes la mayor parte del speedup.
sp
Igual que bp. Solo aparece en el thunk al cruzar la frontera.

Patrón ④ — AOT llamando a BP no AOT-ed

Si el AOT necesita llamar a una función BP que no está AOT-ed (caso raro pero posible), usa un helper:

int32_t r = vm_call_bp(vm, target_pc, arg0, arg1, ...);
// Internamente: prepara frame BP (push saved pc=ret-to-helper,
// bp, cs; push args), ejecuta interp hasta RET, recoge ret del top.

Si el target es desconocido en codegen (función polimórfica vía param), el AOT usa este helper. Si el target es conocido y AOT-ed, C call directa (patrón ③).

Por qué este diseño

Es lo que hacen HotSpot, V8, LuaJIT, todos los JITs serios. Mantener la AOT en C-ABI nativa permite que el compilador C haga register allocation, tail-call optimization, y inlining de los vm_op_* helpers static inline. Si forzásemos la AOT a usar la convención BP (sp/bp por memoria), perderíamos la mayor parte del speedup que mostró el bench (#159: 53×).

Calling convention (CALL / RET / INVOKE_VIRTUAL)

CALL relAddr / CALL_EXT idx

Preconditions: stack tiene los args en orden (último arg en top).

push pc                ; saved pc del caller
push bp                ; saved bp del caller
push cs                ; saved cs del caller
bp ← sp                ; nuevo bp = top
pc ← cs + relAddr      ; (CALL) o targetAbs (CALL_EXT)
cs ← targetCS          ; sólo CALL_EXT lo cambia

Layout del frame tras CALL + ENTER N

              ↑ direcciones altas
  [bp + N-4]  último slot de los locales de ENTER N
  [bp + 0]                                  ← inicio de locales, sp justo tras ENTER
  ┌─────────────────────────┐
  │ saved_cs    [bp - 4]    │
  │ saved_bp    [bp - 8]    │
  │ saved_pc    [bp - 12]   │
  ├─────────────────────────┤
  │ arg_N       [bp - 16]   │
  │ ...                     │
  │ arg_0                   │  ← bp - 12 - paramsCount*4
  └─────────────────────────┘
              ↓ direcciones bajas

RET k

sp -= 4
returnValue = mem[sp]              ; pop del top del callee
pc = mem[bp - 12]
bp_old = mem[bp - 8]
cs = mem[bp - 4]
sp = bp - 12 - k*4                 ; descarta saved + args
bp = bp_old
mem[sp] = returnValue              ; push retVal
sp += 4

INVOKE_VIRTUAL slot, numArgs (L2 v3, fall-back al parent)

this = peek(sp - 4 - numArgs*4)
if this == 0: throwBpRuntimeError("null receiver")
classPtr = mem[this]
desc = classPtr
loop:
  nMethods = readU16(desc + 2)
  bitmapW  = readU16(desc + 4)
  vtBase   = desc + 12 + 2*bitmapW*4
  if slot < nMethods:
    methodOff = readI32(vtBase + slot*4)
    if methodOff != -1: targetCS = getCSForDataAddr(desc); goto found
  parentOff = readI32(desc + 8)
  if parentOff == 0: throwBpRuntimeError(...)
  curCS = getCSForDataAddr(desc)
  desc = curCS + parentOff
  goto loop
found:
  push saved pc, bp, cs ; bp ← sp ; pc ← targetCS + methodOff ; cs ← targetCS
