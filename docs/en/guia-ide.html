BasicPlus — IDE guide

BpIde is the BasicPlus development environment: editor, compiler, local and remote execution, board explorer and debugger — in a single jar with no installation.

1. Starting the IDE

java -jar BpIde-1.0-SNAPSHOT-shaded.jar

There is no installer and no prior setup: the jar carries the compiler, the VM client and this manual (Help → Manual, F1).

2. The window

Four areas:

  • Project tree (top-left): sources, resources, compiled output and dependencies of the open project.
  • Editor (top-right): tabs with BP syntax highlighting; the current line is marked and the position shows in the status bar.
  • Board explorer (bottom-left): the backend selector (serial board or Java VM over TCP), the port, the Connect button and — once connected — the micro's file tree with its button bar.
  • Console and debugging (bottom-right): the Consola, Errores, Variables, Properties módulo, Call Stack and Breakpoints tabs. Below, the / > prompt of the micro command console (§7).

3. Projects

Project → New Project creates the standard layout; File → New adds loose files. A project is described by its .bpbuild/.bpproject (JSON, see the reference §16) and organized as:

  • Sources — the project's .bp files (in the screenshot: App.bp and Util.bp).
  • Resources — data files that Run copies to the micro as-is (configuration, tables…).
  • Output — what the compiler produces: .mod (bytecode) and .bpi (interfaces).
  • Dependencies — resolved imported modules.

4. Editing and compiling

Run → Compile compiles the file or the project; diagnostics show up in the Errores tab with their file, line, column, kind (lexical / syntactic / semantic) and message, separating errors from warnings. Double-clicking a row jumps to that position in the editor.

The compiler does not stop at the first error and does not drown in cascades: it recovers at statement/function level and keeps going, so a single pass surfaces the real errors of the whole module. In the screenshot, a lexical+syntactic error (a = where a := belongs — the language assigns with :=) coexists with an undeclared identifier in another function further down: the first failure does not hide the second. Spurious cascades are pruned (an already-faulty operand produces no derived errors) and, if there were very many, they are capped with an "and N more".

5. Running (and stopping)

  • Run → Run: runs on the PC, on the Java VM (local daemon). The program output goes to the Console.
  • Run → Run on Device: uploads the .mod to the connected board (with any missing dependencies and the project's resources) and runs it there; the output streams back live to the Console.
  • Run → Stop (Ctrl+F2): aborts the running program — local or on the board — without resetting anything. On the board it is a cooperative KILL: the VM dies cleanly and the micro keeps responding.

If the program asks the user for data with IO.prompt, the IDE pops a form and execution pauses until you fill it in: text fields, numeric fields, checkboxes and drop-downs as declared by the program.

IO.prompt is a development utility of the Java VM (PC): the form is drawn by the IDE. It is not available on the micros — there IO.prompt throws a catchable RuntimeError, so wrap it in try/catch if your program must also run on the board.

5.1 native (AOT): installing the ARM toolchain

If the program uses function native, on Run on Device the IDE compiles those functions to machine code (.mdn) and uploads it next to the .mod — on the ARM boards (RP2350 and STM32). That step requires the bare-metal ARM toolchain (arm-none-eabi-gcc) installed on the PC:

  • Windows — the Arm GNU Toolchain (arm-none-eabi) installer from developer.arm.com. If you already use STM32CubeIDE or the pico-sdk, you already have one.
  • Linux — sudo apt install gcc-arm-none-eabi (Debian/Ubuntu) or your distribution's equivalent package.

The IDE autodetects it (the standard Arm GNU Toolchain install location, or the PATH). If it cannot find it, set it manually in Project → AOT (toolchain)…: the path to arm-none-eabi-gcc and the bpgenvm-c root (it provides the runtime headers), each with its own Detectar button. That setting is per machine; which projects use AOT (and for which board) lives in the project's .bpbuild (the "aot" section).

The toolchain is optional: without it, the IDE warns and the program runs interpreted — same behavior, less speed on compute kernels. On the ESP32s (S3 and P4) native runs interpreted in this version.

6. Connecting to the board

In the Explorer: pick the serial board (v1) backend, the micro's COM port and press Connect. The tree shows the device's internal filesystem:

  • /app — your programs (uploaded .mod).
  • /lib — the standard library embedded in the firmware (it reinstalls itself on every boot).
  • /sys — configuration: board.json (board identity), device.json, auto.txt (autorun, §10).

The button bar operates on the selected file: Refresh, Upload, Run, Download, Edit (opens an editor over the micro's file — also via double click), Delete, Save (persists the FS to flash), Log/Clr Log (the firmware's persistent log), Reset and Info (§8). The panel's bottom bar summarizes the FS: files, used and free bytes.

The same panel serves the Java VM (TCP v1): point it at a local or remote daemon and you get the same tree and the same buttons — the protocol is identical.

7. The micro console

The / > prompt under the Console talks directly to the connected device. help lists the commands: 

dir [path] · cd <path> · type <file> · edit <file> · run <file> · del <file>
kill · autorun [file|off] · mem · save · log · reset · cls · help

Two of them deserve a highlight: kill aborts the running program (equivalent to Run → Stop) and autorun manages the standalone startup (§10). If you connect while a program is running, the explorer will report the board as busy — kill takes back control.

8. Device INFO

The Info button interrogates the firmware. In the screenshot, an Adafruit Metro RP2350: the micro and its variant (RP2350B detected at runtime via board.json — 48 GPIO, 24 PWM outputs, 8 ADC channels), the unique serial, frequency and temperature, the real flash measured via JEDEC (16 MB), the SRAM, the detected and self-tested PSRAM (8 MB, used as the VM heap), FS usage and uptime.

9. Debugging

The debugger works the same against the local VM and inside the micro (Debug on Device):

  • F9 — toggle a breakpoint on the line.
  • Debug → Debug / Debug on Device — start in debug mode.
  • F10 step over · F11 step into · Shift+F11 step out · Ctrl+F5 stop.
  • While paused: Variables (locals by name), Properties módulo, Call Stack and the Breakpoints list.

In the screenshot, execution is paused: the breakpoint line is marked, the current line highlighted, and the Variables tab shows the locals by name with their type and value at that instant (here led and i inside the loop). The workflow is identical whether you debug on the local VM or on the micro itself.

10. Standalone device

With your program already on the board: autorun MyApp in the console (it writes /sys/auto.txt and persists it) and reset. From then on the micro starts your program by itself, no PC. The IDE can still connect while the program runs: kill stops it and autorun off removes it — no reflashing ever needed. Full detail in the quick start (Spanish).
