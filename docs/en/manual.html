BasicPlus — Language manual

A structured, typed, module-oriented language. A Java frontend compiles to its own bytecode (the .mod v5 format, "MOD5"), executed by the bpgenvm VM (mark-and-sweep GC, deterministic owner handling, typed exceptions).

1. Introduction

BasicPlus (BP) takes the friendly syntax of Basic / Pascal and combines it with modern semantics: classes with single inheritance, modules as Modula-2-style units, interfaces as contracts, exception handling, and deterministic "ownership" for immediate release of resources.

The overall pipeline is:

// 1) Compilation: each module → a .mod (binary) and a .bpi (textual interface)
java -jar basicplus-frontend.jar samples/hola.bp --compile out --backend=mivm

// 2) Execution: the VM loads the root .mod; the .mod files of its imports are
//    found by the convention ..mod or via 'from "..."'
java -jar bpgenvm.jar out/Hola.mod

2. Hello world

module Hola

  function main(arg: string)
    print "hello, " + arg
  end main

end Hola

The minimum: a module, a function main(arg: string), and matched ends. The VM passes an empty string to main when there are no arguments.

3. Lexical structure

3.1 Comments

Line comments only, with // to the end of the line.

3.2 Identifiers

A letter or _ followed by letters, digits or _. Case-sensitive. Names starting with a double underscore (__name) are reserved by convention for the compiler and the stdlib (e.g. __init, __newRefArray).

3.4 Literals

Any numeric literal may include a visual separator _ between digits (Python-style); the value is identical to the one without it. It works in decimal, hex, binary, the fractional part and the exponent: 1_000_000 · 0xDE_AD · 0b1111_0000 · 2_000_000L · 1_234.5 · 1_0e2.

3.5 Line continuation

A newline ends the statement, except inside open parentheses or brackets: there the newline is not significant and the expression continues on the next line. To split a long expression, wrap it in ( ): 

print ("total = " + intToString(a)
    + " / " + intToString(b))      // OK: inside ( )

var v: integer[] := [1, 2,
                       3, 4]          // OK: inside [ ]

4. Types

4.1 Primitives

TypeRange / detail
integerSigned 32-bit integer (-2³¹..2³¹-1)
longSigned 64-bit integer. Literal with L suffix: 4000000000L
floatIEEE-754 32-bit
doubleIEEE-754 64-bit. Literal with d suffix: 1.5d
byteNarrow unsigned integer, 8 bits (0..255)
int8Narrow signed integer, 8 bits (-128..127)

The narrow types (byte/int8/ word/int16) are storage types: in arrays and fields they occupy 1–2 bytes per element; on the stack and in expressions they operate as integer (a read sign-extends or zero-extends according to the type). Assigning an out-of-range literal is a compile error, with the exact range in the message.

4.1.1 Promotion and casts

In mixed operations and assignment there is automatic promotion to the widest type: integer → long → float → double (the narrow types are promoted to integer first). The opposite direction is NEVER implicit: narrowing requires an explicit cast using the call syntax of the target type: 

var d: double := 3.9d
var i: integer := integer(d)     // 3 — truncates toward zero (integer(-3.9d) = -3)
var L: long    := long(i)        // explicit widening is available too
var f: float   := float(L)
var b: byte    := byte(i)        // validates the RANGE at runtime:
                                  // outside 0..255 throws RuntimeError (catchable)

The numeric casts are integer(x), long(x), float(x), double(x) and the narrow ones byte(x), int8(x), word(x), int16(x). integer(x) on a float/double truncates toward zero; to round, use floor, ceil or round. Casting a literal outside the target type's range is rejected at compile time.

4.2 Arrays

An array is an indexable sequence of elements of the same type. BP distinguishes two variants depending on whether the size is known at declaration time, and that determines where the storage lives.

4.2.1 Fixed-size arrays  T[N]

The size is an integer literal > 0 known at compile time, and the elements are primitives (integers, narrow types, long, float, double, boolean — for string or classes use T[]). The variable IS the array: it has a fixed binding (it cannot be reassigned; its elements are writable, though). Where it lives depends on the scope:

ScopeWhere it lives
Module globalBaked into the data block of the .mod (including the literal initializer): zero heap and zero startup code.
Function localAn N-element heap array, zero-allocated when the declaration is entered. Managed by the GC like any array.
Class fieldNot supported (compile error): declare T[] and initialize it in the constructor with newIntArray/newByteArray/newLongArray.
// Module: baked into the data block; the literal initializer may be shorter
// than N — the rest stays at zero. [1,2,250] fits in byte[4] (it is validated
// element by element, not by array assignability).
var tabla: integer[4] := [10, 20, 30]   // → 10,20,30,0
var mascara: byte[4] := [1, 2, 250]

function Main()
  // Local: N elements at zero on the heap; len() and iteration work
  var buf: byte[64]                    // typical I2C/SPI buffer
  buf[0] := 42
  print len(buf)                       // 64
  // buf := other    ← error: fixed binding (write to its elements instead)
end Main

A local fixed array is a normal heap array: it can be passed to a function that takes T[] and disappears via GC once it is no longer referenced. The module one lives in the data block and is never freed. There are no const arrays: a module const only accepts primitive literals (see §5.2).

4.2.2 Variable-size arrays  T[]

The size is decided at run time. The array is allocated on the heap (opcode NEWARRAY) and referenced through a ref (an i32 holding the pointer to the block). Managed by the GC. 

// From a literal: the frontend emits NEWARRAY n + an ASTORE for each element
var primos: integer[] := [2, 3, 5, 7, 11]

// From a function: split, listDir, etc. return arrays
var partes: string[] := split("a,b,c", ",")
var ficheros: string[] := listDir(".")

The VM uses five physical heap types depending on the element width: TYPE_ARRAY_I8 (byte/int8 and UTF-8 strings), TYPE_ARRAY_I16 (word/int16), TYPE_ARRAY_I32 (integer; also floats, as 4 opaque bytes), TYPE_ARRAY_I64 (long/double, 8 bytes per element) and TYPE_ARRAY_REF (4 bytes, object refs, scanned by the GC). The array carries its own length as the first slot. For the narrow widths, a read extends the value to i32 (signed or unsigned according to the type) and a write truncates to the element width.

4.2.3 Access and iteration

Indexed with arr[i] (0-based). for-in iterates over all elements (see §7.3). 

for n in primos do
  total += n
next n
For dynamic collections (grow, shrink, mixed types), use List or OwnerList instead of T[]. T[] arrays have a fixed size after creation (there is no append); List resizes internally.

4.3 Enums

public enum Dia
  LUNES               // = 0
  MARTES              // = 1
  MIERCOLES
  JUEVES
  VIERNES
  SABADO := 10          // explicit jump
  DOMINGO             // = 11
end Dia

var d: Dia := Dia.MIERCOLES

Enums are symbolic names for integer constants. They auto-number from 0 unless assigned explicitly.

4.4 Classes

Any class Foo ... end Foo introduces a type Foo usable as a variable, parameter and return type. Refs admit null.

4.5 any

An internal type used by some stdlib signatures (e.g. List.add(item: any)). It is bidirectionally assignable: any ref goes in, and any target type accepts a value of type any. You cannot write any as a type in BP source.

5. Declarations

5.1 var

var nombre: string
var edad:   integer := 30
var total:  long := 0L
var media:  double
var x, y, z: integer                  // multiple names
var pts: integer[100]                  // fixed array (§4.2.1)
var p: Punto := Punto(3, 4)             // class ref

At module level, a var with a literal initializer (numeric/boolean/string) is baked into the data block of the .mod: it generates no startup code and the variable remains reassignable. A non-literal module initializer is ignored with a warning — assign it in the module's initializer function (the function with the same name as the module).

5.2 const

public const PI: float := 3.14159
public const MAX_MS: long := 86400000L
const SALUDO: string := "hello"
const DEBUG: boolean := false
const MIN: integer := -40             // '-' on a literal, OK

A module const requires a primitive literal (integer/float/long/double/boolean/string, with an optional -) — the value is inlined at each use, occupies no slot, and public ones travel in the .bpi so importers inline them too. For a computed value or a non-primitive type, use a var assigned in the module's initializer function. It is not assignable in any scope, and there are no const arrays (§4.2.1).

5.3 var owner

var owner r: Recurso := Recurso("db")
// when it goes out of scope or is reassigned, the object it points to is freed
// deterministically via FREE_REF (including the owner cascade).

Full detail in §12 Owner semantics.

7. Control flow

7.1 if / elseif / else

if edad < 18 then
  print "under age"
elseif edad < 65 then
  print "adult"
else
  print "senior"
endif

7.2 while / do

while i < n do
  total := total + i
  i := i + 1
endwh

do
  i := i + 1
while i < 100          // do-while: condition after the body

7.3 for

Two variants: numeric and foreach (for-in).

// Numeric: i := from to to [step]
for i := 0 to 9 do
  print i
next i

for i := 10 to 1 step -1 do
  print i
next i

// Foreach over an array (fixed or variable, any element type)
var primos: integer[] := [2, 3, 5, 7, 11]
for n in primos do
  total += n
next n

The body closes with next [name] (not endfor); the name after next is optional and, if present, must match the iteration variable (documentation use).

7.4 switch

switch dia
  case Dia.SABADO, Dia.DOMINGO
    print "weekend"
  case Dia.LUNES
    print "back to work"
  default
    print "weekday"
endsw

7.5 break / continue

Breaks out of (or jumps to the next iteration of) the nearest enclosing loop (while, do, for, switch).

7.6 parallel / endpar

Syntactic sugar for starting several threads without writing a subclass of Thread for each one. Each case introduces a thread whose body is the block's body; the default (optional) runs on the calling thread between the start()s and the join()s; after endpar all threads are joined. 

function main(arg: string)
  parallel
    case t1 := Thread(2048)
      var r: integer := sumaRango(1, 50000)
      print "t1 = " + intToString(r)

    case t2 := Thread(2048)
      var r: integer := sumaRango(50001, 100000)
      print "t2 = " + intToString(r)

    default
      // runs on the calling thread; sees its locals and globals
      print "default while t1/t2 work"
  endpar
  print "all joined"
end main

Conceptually equivalent to the long pattern (class W extends Thread …; W().start(); W().join()), but the compiler synthesizes an anonymous subclass of Thread for each case, with the case body as its run(). The variables t1, t2, … are visible in the enclosing scope after endpar: they are already joined and can be inspected.

The first token after := must be literally Thread with an integer argument (the worker's stack). For thread-owned state with fields, use the classic Thread-subclass pattern.

Visibility rules for the case body (object encapsulation — see 14.3):

SymbolVisible
Locals declared in the body itself✓
stdlib (print, intToString, yield(), Mutex, ...)✓
Module classes, functions, consts✓
Module global variables (var)✗ semantic error
Locals of the enclosing scope✗ "unresolved identifier"

The default block, in contrast, runs in the enclosing scope and sees EVERYTHING: locals, globals, etc. For branches to coordinate, they share state via objects (Mutex, List, shared classes).

8. Functions

public function suma(a: integer, b: integer): integer
  return a + b
end suma

function imprimir(msg: string)        // no ': T' → void
  print msg
end imprimir
  • public → exported in the .bpi, callable cross-library.
  • By default, functions are private to the module.
  • The return type is declared with : T after the parentheses. If omitted, the function is void.
  • Returning is done only with return [value]. With no explicit return, void.

8.1 Default parameter values

public function label(text: string, prefix: string := "[", suffix: string := "]"): string
  return prefix + text + suffix
end label

label("hi")              // "[hi]"
label("hi", "<")         // "<hi]"
label("hi", "<", ">")    // "<hi>"

A parameter with := value is optional at the call site. Rules:

  • The default value must be a constant literal (numeric, string, boolean).
  • Parameters with defaults go at the end of the list; arguments are filled by position.
  • The default is substituted at the caller at compile time (zero cost in the VM) and travels in the .bpi: it works the same way when calling functions from another library.

8.2 Tuples (multiple return)

// A function can return SEVERAL values: a tuple type (T1, T2, ...)
function divmod(a: integer, b: integer): (integer, integer)
  return (a / b, a mod b)
end divmod

// Unpacking with { ... } := onto already-declared lvalues
var q: integer := 0
var r: integer := 0
{ q, r } := divmod(17, 5)        // q=3, r=2

Tuples are first-class: besides being unpacked on the fly, they can be stored in a variable, passed as a parameter and returned in stored form. The unpacking targets can be any lvalues, not just variables: 

var t: (integer, string) := pair()   // store the tuple
{ n, s } := t                          // ... and unpack it later

{ arr[0], arr[1] } := coords()        // into array elements
{ p.x, p.y } := coords()               // into object fields

lista.add(pair())                      // go into collections via any
var t2: (integer, string) := lista.get(0)

A tuple is not printed directly (print t is an error): unpack it first. The targets of { ... } := must already exist (it does not declare variables).

8.3 native functions (AOT)

A function marked native function is a candidate for AOT compilation (Ahead-Of-Time): besides the normal bytecode, the compiler can translate it to native C for the micro, which runs directly (without the interpreter loop) to speed up hot compute kernels. 

native function sumaCuadrados(n: integer): integer
  var s: integer := 0
  var i: integer := 1
  while i <= n do
    s := s + i * i
    i := i + 1
  endwh
  return s
end sumaCuadrados
  • Additive: the .mod ALWAYS contains the function's bytecode. The AOT is an optional overlay (.mdn) that the runtime uses if it is present and matches the micro's architecture; otherwise the function runs interpreted. Not compiling the AOT never breaks the program.
  • Per-platform behavior:
    Platformnative function
    Raspberry Pi Pico 2 / Metro (RP2350, ARM Cortex-M33) AOT-accelerated (≈40–90× on integer kernels).
    STM32 (Nucleo-U575, ARM Cortex-M33) AOT-accelerated — same Thumb-2 .mdn (≈45× measured on an LZSS kernel on-board).
    ESP32-S3 (Xtensa LX7) Interpreted — same semantics, no speedup. The .mdn is ARM code; AOT for Xtensa is future work.
    PC VM (host) Interpreted (there is a host AOT path for testing only).
  • Supported subset: leaf functions over primitives, arrays (integer[], byte[], …), string, module variables and builtins (print, now, …). A runtime failure (index out of range, null reference) or a throw RuntimeError("…") inside the native propagates to the BP try/catch that wraps the call. What still does NOT fit (calling object methods, try/catch inside the native) falls back to interpreted bytecode automatically. Local fixed arrays type[N] are not supported inside a native (compile error): use newIntArray/newByteArray/newLongArray.
  • Calling normal BP functions from a native is possible (the bridge exists), but it crosses back into the interpreter: the speed gained is lost on every crossing. The compiler detects this and emits a performance warning.
  • How the .mdn is generated: the IDE does it automatically on Run on Device (v3, on ARM boards: RP2350/STM32): it detects the ARM bare-metal toolchain (arm-none-eabi-gcc — a standard install of the Arm GNU Toolchain or on the PATH; set manually in Project → AOT (toolchain)…; installation: IDE Guide §5.1), emits C, compiles Thumb-2 PIC and uploads the .mdn alongside the .mod. If there is no toolchain, it warns and the program runs interpreted (the .mdn is optional and additive). The step can also be done by hand with the build_mdn.sh script (AotMain emits C → arm-none-eabi-gcc → MdnPack packages it; it looks for the compiler on the PATH or in the GCC environment variable).

9. Classes

public class Persona
  public var nombre: string
  public var edad:   integer

  public function Persona(nombre: string, edad: integer)
    this.nombre := nombre
    this.edad   := edad
  end Persona

  public function saludar(): string
    return "Hi, I'm " + this.nombre
  end saludar
end Persona

9.1 Fields

A var inside the class declares an instance field. public visibility is optional. Access via this.field from inside; obj.field from outside (only if public).

A field does NOT allow a fixed-size array type[N] (compile error): declare type[] and initialize it in the constructor with newIntArray(N)/newByteArray(N)/newLongArray(N). Fields are not exported in the .bpi — they are private to the owning module by OO design; expose a property if you need cross-module access.

9.2 Methods

Like functions but inside the class. The implicit parameter this points to the instance. All non-static methods are virtual (dynamic dispatch via vtable).

9.3 Constructor

A function with the same name as the class. It runs when you do ClassName(args). When super(args) is called, it must be the first statement.

Implicit super() (Java model). If the class has a base and its constructor does not call super(...), the compiler injects the call to the parent's no-argument constructor. If the parent's constructor requires arguments, the explicit call is mandatory (compile error if missing). If the parent has no constructor, there is nothing to call and nothing is injected.

9.4 Inheritance

public class Animal
  public var nombre: string
  public function Animal(nombre: string)
    this.nombre := nombre
  end Animal
  public function hablar()
    print "..."
  end hablar
end Animal

public class Perro extends Animal
  public function Perro(nombre: string)
    super(nombre)
  end Perro
  public function hablar()
    print this.nombre + " says woof"
  end hablar
end Perro

Single inheritance. super.method(args) explicitly invokes the parent's method inside an override. Inheritance also works cross-module: the base class can come from another imported library (vtable, super(...), instanceof and statics included). x instanceof Foo checks the dynamic type at runtime (including the Exception hierarchy).

9.5 Properties

public class Punto
  var _x: integer
  public property x: integer
    get
      return this._x
    endget
    set(v)
      if v < 0 then throw RuntimeError("negative") endif
      this._x := v
    endset
  endprop
end Punto

// Short form: backing field + trivial getter/setter auto-generated
public property y: integer

A property synthesizes a backing field and two methods getX/setX. From code it is accessed like a field (p.x, p.x := 5).

sync properties (thread-safe)

public class Account
  public sync property balance: integer
  public function Account(initial: integer)
    this.balance := initial
  end Account
end Account

The sync modifier before property wraps the auto-generated getter and setter with lock() / unlock() against a Mutex shared per instance. The compiler:

  • Automatically declares a hidden field __syncMutex (inherited by subclasses — it is declared only once, in the first link of the hierarchy that needs sync).
  • Initializes it with Mutex() on entering the constructor. If the class has no user constructor, it synthesizes a default one that only performs that initialization.
  • Inserts this.__syncMutex.lock() at the start and this.__syncMutex.unlock() at the end of each getX/setX.

Guarantees: each individual access to the backing field is atomic with respect to other threads. A read never returns a partial state; a write is indivisible.

Important: a compound expression such as this.value := this.value + 1 is NOT atomic as a whole — it decomposes into get + add + set, each individually atomic, but another thread can slip in between them. For compound atomic operations, use an explicit Mutex around the block.

A return inside the custom body of a sync get/set correctly releases the mutex: the emitter injects the unlock before the jump to the exit point (no explicit try/finally is needed in BP). The same applies to module-level sync properties.

Module-level properties

module Counter
  // A public module property. The state lives in a private
  // global, accessible from outside ONLY through the get/set.
  public property tick: integer
  public function bump()
    tick := tick + 1
  end bump
end Counter

module App
  import Counter
  function main(arg: string)
    Counter.bump()
    print Counter.tick           // cross-module read → 1
    Counter.tick := 42          // cross-module write
  end main
end App

BP also lets you declare a property outside a class, at module level. It is the canonical way to expose mutable state to other modules: module vars are private by design (they are not exported in the .bpi), but a public property exposes its synthetic getter/setter, accessible via Module.name from another module that imports it.

Internally the compiler emits:

  • A private global __prop_<name> that is the backing field.
  • A public function __prop_get_<name>() that returns the value.
  • A public function __prop_set_<name>(__val) that writes it.
  • In the .bpi, a line prop <name>:<type> public that travels to importing modules (without exposing the accessors as normal functions).

The identifier tick within the module itself resolves to the getter/setter (not the global directly), so encapsulation is respected inside the owning module too. If you need to initialize the property with a value other than zero, do it in the module's initializer function (the function with the same name as the module), just as with a module-level var.

sync is also valid on module properties: 

module Stats
  public sync property contador: integer
end Stats

In this case the module declares a single global Mutex (__prop_mutex) shared by all its sync properties. It is initialized when the module loads (in __init) and the get/set wrap their body with lock/unlock. The guarantees and limitations are the same as for sync in classes.

MVP limitations:

  • Module-level owner properties are not supported (classes only).
  • final is not supported at module level (the modifier applies only to class properties).
  • The property's init expression is ignored at compile time; use the module's initializer function to set the initial value.

9.6 Static members / final

BP has no static keyword: a member is static when it is declared with a name qualified by the class (Class.member) inside the class body. It works for const, var, property and function: 

public class Counter
  public const Counter.MAX: integer := 100   // static const
  public var Counter.viva: integer            // static var

  public property Counter.total: integer     // static property (auto get/set;
                                               // custom get/set allowed with `field`)

  public function Counter.bump()               // static method
    Counter.total := Counter.total + 1
  end Counter.bump
end Counter

// Access is ALWAYS qualified, with no instance — also from instance
// methods and from other modules (they travel in the .bpi):
Counter.total := 7
Counter.total += 3
Counter.bump()

Static state lives as a global of the owning module (one per class, not per instance); 8-byte types (long/double) work the same. final on a class: not extensible; on a method: not overridable (the modifier goes before function or class).

9.7 Fields owner

public class Caja
  public var owner contenido: Recurso
  public function Caja(r: Recurso)
    this.contenido := r
  end Caja
end Caja

When a Caja is freed (via FREE_REF), its owner field contenido is freed too, recursively.

10. Modules

10.1 library (optional)

library "com.example.app"
module Util
  ...
end Util

If you declare library, the artifact is named com.example.app.Util.mod and .bpi. Without library, it is called Util.mod.

10.2 main(arg: string) and module constructor

module MiApp
  // Initializer (auto): runs when the module loads, after the imports.
  function MiApp()
    print "[init] loaded"
  end MiApp

  function main(arg: string)
    print "running with arg='" + arg + "'"
  end main
end MiApp
  • function ModuleName() is the module "constructor". It runs after all imports have been initialized.
  • function main(arg: string) is the entry-point. Only the root module's main is called; the mains of dependencies remain as a utility for standalone testing.
  • Legacy form: function Main() with no parameters (still accepted).

10.3 import / from

module App
  import Util                       // same library
  import com.example.Util           // qualified path
  import ext.tools.Helper from "plugins/helper.mod"
  ...
end App
  • The last segment of the path is the local alias (Util, Helper). You use it as Util.foo().
  • from "<path>" tells the VM where to find the .mod at runtime. Compilable when the frontend locates the source .bp by deriving the path.

10.4 module interface

library "com.example"
module interface LogApi
  public function log(msg: string)
  public function level(): integer
  public const VERSION: integer := 1
end LogApi

An interface is a contract (a Modula-2 DEFINITION MODULE): public signatures only, no function bodies. It compiles to a .bpi with the flag interface true and produces no .mod.

Interfaces can extend another interface (contract inheritance): 

module interface LogApiV2 extends com.example.LogApi
  public function logWithLevel(lvl: integer, msg: string)
  public function flush()
end LogApiV2

10.5 implements

module ConsoleLogger implements com.example.LogApi
  public function log(msg: string)
    print "[CON] " + msg
  end log
  public function level(): integer
    return 0
  end level
  public const VERSION: integer := 1
end ConsoleLogger

The analyzer verifies that the module exposes ALL the contract's signatures (functions with identical types, consts with the same type, enums with the same members and values). The impl may freely add public extras not required by the contract. Typical errors:

  • does not implement the function 'foo' required by 'I'
  • function 'foo': declared return 'float', 'integer' was expected
  • function 'foo' required by 'I' must be public

10.6 Binding import I:Impl

module App
  import com.example.LogApi:ConsoleLogger
  // Equivalent with from:
  // import com.example.LogApi:ConsoleLogger from "plugins/logger.mod"

  function main(arg: string)
    LogApi.log("hi")          // types against LogApi; CALL_EXT to ConsoleLogger.log
  end main
end App

Compile-time: the LogApi namespace is typed against the interface. Codegen: each LogApi.x emits CALL_EXT to the bound impl. Switching impl = editing the binding line and recompiling.

LSP compatibility: import LogApi:BufferedLogger is valid even if BufferedLogger declares implements LogApiV2 and LogApiV2 extends LogApi. The driver walks the implements → extends chain.

10.7 Class ↔ module encapsulation

From inside a class, only public module symbols are accessible (functions, consts, vars, properties). Types (classes, enums, import namespaces) are always visible. 

module M
  function helperPriv(): integer     // private
    return 99
  end helperPriv

  class C
    public function bad(): integer
      return helperPriv()             // ERROR: not public
    end bad
  end C
end M

11. Exceptions

try
  riskyOp()
catch e: RuntimeError
  print "runtime: " + e.msg
catch e: MiError
  print "my error: " + e.detalle
catch e
  print "unknown"
finally
  print "cleanup"
endtry

throw RuntimeError("something failed")
  • try installs a handler on a VM handler stack.
  • catch e: Foo only catches instances assignable to Foo. catch e (no type) catches any exception.
  • finally always runs (with or without an exception).
  • throw expr throws the value. If it is a class, its class_ptr is used for the typed catch match.
  • All exceptions descend from Exception (with its msg property), which lives in the standard module Core along with RuntimeError. The import Core is IMPLICIT: just using try/throw or naming Exception/RuntimeError makes them available.
  • A user exception is a class that extends Exception (it inherits msg; add whatever fields you want). throw and the typed catch require classes that descend from Exception.
  • The VM's native errors (division by zero, null reference, index out of range, failures in native functions...) are thrown as RuntimeError and caught the same way, regardless of the module where they occur.
  • An uncaught exception terminates the VM, showing a stack trace with the symbolic name of each frame.

12. Owner semantics

BP supports deterministic object destruction as a complement to the traditional GC. The basic rule: an object can have one owner; when the owner is freed, the owned object is freed too (cascade).

12.1 var owner x: T

  • On assigning x := y, the previous value of x is freed (FREE_REF) beforehand.
  • On leaving scope, the current value of x is freed.
  • Assigning x to a non-owner variable copies the ref without transferring ownership (it still belongs to x).
  • Assigning to another owner variable transfers: the source is left at null without being freed.

12.2 Fields owner

public var owner f: T inside a class. The class descriptor holds a bitmap of owners; when the instance is freed, each marked slot is freed recursively. The opcode SET_FIELD_OWNER frees the field's previous value before writing the new one (an owner-safe atomic assignment).

12.3 Cascade through ref arrays

If an owner field holds a reference to a TYPE_ARRAY_REF array, freeing the container also frees all the array's non-null slots. This is the mechanism behind OwnerList.

14. Concurrency

The VM simulates a 2-core microcontroller: by default it launches N real Java workers (default N=2; configurable with --workers=N, see §15.2) that share out the queued BP threads.

14.1 Execution model

  • The main BP thread (tid 0) starts first: it runs the module's initializer and then main(arg). It has a large stack (16 KB). Only it can execute the HALT instruction that shuts the VM down.
  • Any subclass of Thread adds a new BP thread when you call start(). The VM assigns it a stack (default 2 KB) and queues it RUNNABLE. A free Java worker picks it up and runs its run() until return (THREAD_EXIT) or yield()/sleep()/join()/a lock.
  • Each BP thread has its own context (ThreadContext): PC, SP, BP, CS, stack region, exception-handler stack. The heap (including objects owned by threads) is shared.
  • When all BP threads finish, the VM shuts down. If main does HALT first, the in-flight workers are cut off at their next opcode.

14.2 Preemption and yield()

A daemon timer (ScheduledExecutorService, default quantum 10 ms) sets yieldRequested=true on every RUNNING BP thread. The interpreter loop observes it between opcodes (a volatile field) and abandons the inner loop with a YIELD signal; the worker re-queues the thread and picks up the next RUNNABLE.

In parallel with preemption, BP code can yield the CPU cooperatively:

  • yield() — yields once; the thread stays RUNNABLE and the scheduler picks it when its turn comes.
  • sleep(ms) — marks BLOCKED_SLEEP until now() + ms. If there is no other RUNNABLE, the Java worker sleeps until the next wake-up.
  • join() on another thread — marks BLOCKED_JOIN until the target finishes.
  • Mutex.lock() on a taken mutex — marks BLOCKED_MUTEX until hand-off.

14.3 Encapsulation of case bodies

The body of each case inside a parallel is the run() of an anonymous Thread subclass synthesized by the compiler. It follows the same object-encapsulation rule as a normal class method: it cannot read/write the module's global variables. It can call module functions, instantiate classes, use consts and call builtins.

To coordinate, threads use shared objects (Mutex, List, stateful classes) instead of mutable globals. The compiler rejects the direct read with the message: 

semantic error: the body of a 'case' inside 'parallel' cannot
access the global variable 'X'; use a const, a function or a
shared object

The default block does NOT suffer this restriction: it runs on the calling thread and sees its locals and globals.

14.4 GC and concurrency

The mark-and-sweep GC is conservative: it scans the stacks of all live threads and the modules' data blocks to identify roots. The heap operations (heapAlloc, gc(), freeOwnedObject) are serialized under vmLock — two workers cannot mutate the heap at the same time.

Stop-the-world safepoint: when a worker needs the GC (because heapAlloc exhausts the heap), it sets a global flag stopTheWorld. The other workers check it at the start of each interpreter iteration: they synchronize tc.{pc,sp,bp,cs} and abandon runOnContext with a YIELD signal. They return to the WorkerLoop, see the flag and park in vmLock.wait(). The worker that requested GC waits until no other is in the RUNNING state (those parked in vmLock.wait inside heapAlloc are counted as parked via parkedInHeapAlloc), runs gcLocked, and then clears the flag.

In addition, each tc keeps an alloc anchor (tc.allocAnchor): heapAlloc leaves the just-allocated ref in it before releasing vmLock. If another worker triggers GC between heapAlloc's return and the caller's subsequent push, the GC scanner sees the anchor as an extra root and does not free the object in transit.

Wraps of allocation opcodes: NEW_OBJECT and the NEWARRAY variants hold vmLock from allocation until publication to the BP stack (tc.sp synchronized). This closes the window in which another worker could sweep the just-created object before its ref reaches the stack of whoever created it.

Known residual (~5–10 % under very high contention): some allocation builtins (allocVmString, allocVmRefArray, GROW_REF_ARRAY) still release vmLock between the allocation and the final push. When a producer/consumer pattern with many simultaneous allocations materializes, occasional corruption can appear. Symptoms: INVOKE_VIRTUAL on null, "Address 0", opcode 0xA6.

When the residual manifests, only the affected thread dies — the rest of the program continues because BpThreadFault prevents a single failure from bringing down the whole VM (see the code in VirtualMachine.WorkerLoop).

  • synctest.bp (class + sync property + 5000 iters × 2 threads): ~90 % pass-rate.
  • synclisttest.bp (SyncList + 1000 prods + 2 consumers): ~93 % pass-rate.
  • modpropsync.bp (module-level sync property): ~95 % pass-rate.
