BasicPlus — Manual de referencia

Lenguaje estructurado tipado, orientado a módulos. Frontend en Java compila a un bytecode propio (formato .mod v5, "MOD5") ejecutado por la VM bpgenvm (mark-and-sweep GC, manejo determinístico de owners, excepciones tipadas).

1. Introducción

BasicPlus (BP) toma la sintaxis amable de Basic / Pascal y la combina con semántica moderna: clases con herencia simple, módulos como units Modula-2-style, interfaces como contratos, manejo de excepciones, y "ownership" determinístico para liberación inmediata de recursos.

El pipeline general es:

// 1) Compilación: cada módulo → un .mod (binario) y un .bpi (interface textual)
java -jar basicplus-frontend.jar samples/hola.bp --compile out --backend=mivm

// 2) Ejecución: la VM carga el .mod raíz; los .mod de los imports los
//    encuentra por convención ..mod o por 'from "..."'
java -jar bpgenvm.jar out/Hola.mod

2. Hola mundo

module Hola

  function main(arg: string)
    print "hola, " + arg
  end main

end Hola

Mínimo: un module, una function main(arg: string), y end emparejados. La VM pasa al main una cadena vacía si no hay argumentos.

4. Tipos

4.1 Primitivos

TipoRango / detalle

Promoción automática integer → float en operaciones mixtas y asignación. No hay coerción float → integer implícita (usar floor, ceil, round).

4.2 Arrays

Un array es una secuencia indexable de elementos del mismo tipo. BP distingue dos variantes según se conozca o no el tamaño en tiempo de declaración, y eso determina dónde se aloja el almacenamiento.

4.2.1 Arrays de tamaño fijo  T[N]

El tamaño se conoce en tiempo de compilación. El espacio se reserva en línea, no en el heap, junto al ámbito donde se declara:

ÁmbitoDónde vive
Local de funciónEn el stack frame de la función (BP+offset).
Global de móduloEn el data block del .mod (CS+offset).
Constante de módulo (con literal)En el data block, valores inmutables.
// Local: 10 enteros reservados en el frame (40 bytes en stack)
var nums: integer[10]
nums[0] := 42
print nums[0]

// Global: el data block del módulo guarda el array entero
var tabla: integer[256]            // fuera de cualquier función

// Constante con valor literal: data block, inmutable
public const PRIMOS: integer[] := [2, 3, 5, 7, 11]

Sin asignación inicial, los elementos parten a cero / null. El array fijo NO se libera porque no vive en el heap; desaparece con su frame / módulo.

4.2.2 Arrays de tamaño variable  T[]

El tamaño se decide en tiempo de ejecución. El array se aloja en el heap (opcode NEWARRAY) y se referencia mediante una ref (i32 con el puntero al bloque). Gestionado por el GC. 

// Desde un literal: el frontend emite NEWARRAY n + ASTORE de cada elemento
var primos: integer[] := [2, 3, 5, 7, 11]

// Desde una función: split, listDir, etc. devuelven arrays
var partes: string[] := split("a,b,c", ",")
var ficheros: string[] := listDir(".")

La VM usa tres tipos físicos en el heap según el ancho del elemento: TYPE_ARRAY_I8, TYPE_ARRAY_I16, TYPE_ARRAY_I32 (también vale para floats: 4 bytes opacos), y TYPE_ARRAY_REF (4 bytes, refs a objetos, escaneado por el GC). El array lleva su propia longitud como primer slot.

4.2.3 Acceso e iteración

Indexado con arr[i] (0-based). for-in recorre todos los elementos (ver §7.3). 

for n in primos do
  total += n
next n
Para colecciones dinámicas (crecen, encogen, mezcla de tipos), usa List o OwnerList en lugar de T[]. Los arrays T[] tienen tamaño fijo tras su creación (no hay append); List redimensiona internamente.

4.4 Clases

Cualquier class Foo ... end Foo introduce un tipo Foo usable como tipo de variable, parámetro y retorno. Las refs admiten null.

4.5 any

Tipo interno usado por algunas signaturas de stdlib (e.g. List.add(item: any)). Es bidireccionalmente asignable: cualquier ref entra, cualquier tipo target acepta un valor de tipo any. No se puede escribir any como tipo en BP source.

7.3 for

Dos variantes: numérico y foreach (for-in).

// Numérico: i := from to to [step]
for i := 0 to 9 do
  print i
next i

for i := 10 to 1 step -1 do
  print i
next i

// Foreach sobre un array (fijo o variable, cualquier tipo de elemento)
var primos: integer[] := [2, 3, 5, 7, 11]
for n in primos do
  total += n
next n

El cuerpo se cierra con next [nombre] (no endfor); el nombre tras next es opcional y, si está, debe coincidir con la variable de iteración (uso documental).

7.5 break / continue

Sale (o salta a la próxima iteración) del bucle más cercano (while, do, for, switch).

7.6 parallel / endpar

Azúcar sintáctico para arrancar varios threads sin escribir una subclase de Thread por cada uno. Cada case introduce un thread cuyo cuerpo es el del bloque; el default (opcional) corre en el thread llamador entre los start()s y los join()s; tras endpar todos los threads están joineados. 

function main(arg: string)
  parallel
    case t1 := Thread(2048)
      var r: integer := sumaRango(1, 50000)
      print "t1 = " + intToString(r)

    case t2 := Thread(2048)
      var r: integer := sumaRango(50001, 100000)
      print "t2 = " + intToString(r)

    default
      // corre en el thread llamador; ve sus locales y globals
      print "default mientras t1/t2 trabajan"
  endpar
  print "todos joineados"
end main

Equivalente conceptual al patrón largo (class W extends Thread …; W().start(); W().join()), pero el compilador sintetiza una subclase anónima de Thread por cada case, con el cuerpo del case como su run(). Las variables t1, t2, … son visibles en el scope encerrador tras endpar: ya están joineadas y son inspeccionables.

El primer token tras := debe ser literalmente Thread con un argumento entero (el stack del worker). Para state propio del thread con campos, usa el patrón clásico de subclase de Thread.

Reglas de visibilidad del cuerpo del case (encapsulación de objeto — ver 14.3):

SímboloVisible
Locales declaradas en el propio cuerpo✓
stdlib (print, intToString, yield(), Mutex, ...)✓
Clases, funciones, consts del módulo✓
Variables globales (var) del módulo✗ error semántico
Locales del encerrador✗ "identificador no resuelto"

El bloque default, en cambio, corre en el encerrador y ve TODO: locales, globals, etc. Para que las ramas se coordinen comparten estado vía objetos (Mutex, List, clases compartidas).

8. Funciones

public function suma(a: integer, b: integer): integer
  return a + b
end suma

function imprimir(msg: string)        // sin ': T' → void
  print msg
end imprimir
  • public → exportada en el .bpi, llamable cross-library.
  • Por defecto, las funciones son privadas al módulo.
  • El tipo de retorno se declara con : T tras los paréntesis. Si se omite, la función es void.
  • Sólo retorno con return [valor]. Sin retorno explícito, void.

9. Clases

public class Persona
  public var nombre: string
  public var edad:   integer

  public function Persona(nombre: string, edad: integer)
    this.nombre := nombre
    this.edad   := edad
  end Persona

  public function saludar(): string
    return "Hola, soy " + this.nombre
  end saludar
end Persona

9.1 Campos

var dentro de la clase declara un campo de instancia. Visibilidad public opcional. Acceso por this.campo desde dentro; obj.campo desde fuera (sólo si es público).

9.2 Métodos

Como funciones pero dentro de la clase. El parámetro implícito this apunta a la instancia. Todos los métodos no-estáticos son virtuales (dispatch dinámico vía vtable).

9.3 Constructor

Función con el mismo nombre que la clase. Se ejecuta al hacer NombreClase(args). Si la clase tiene clase base, debe llamar a super(args) como primera sentencia.

9.4 Herencia

public class Animal
  public var nombre: string
  public function Animal(nombre: string)
    this.nombre := nombre
  end Animal
  public function hablar()
    print "..."
  end hablar
end Animal

public class Perro extends Animal
  public function Perro(nombre: string)
    super(nombre)
  end Perro
  public function hablar()
    print this.nombre + " dice guau"
  end hablar
end Perro

Herencia simple. super.metodo(args) invoca explícitamente al método del padre dentro de un override.

9.5 Properties

public class Punto
  var _x: integer
  public property x: integer
    get
      return this._x
    endget
    set(v)
      if v < 0 then throw RuntimeError("negativo") endif
      this._x := v
    endset
  endprop
end Punto

// Forma corta: backing field + getter/setter triviales auto-generados
public property y: integer

Una property sintetiza un campo respaldo y dos métodos getX/setX. Desde código se accede como un campo (p.x, p.x := 5).

Properties sync (thread-safe)

public class Account
  public sync property balance: integer
  public function Account(initial: integer)
    this.balance := initial
  end Account
end Account

El modificador sync antes de property envuelve el getter y setter auto-generados con lock() / unlock() contra un Mutex compartido por instancia. El compilador:

  • Declara automáticamente un field oculto __syncMutex (heredado por subclases — sólo se declara una vez en el primer eslabón de la jerarquía que necesita sync).
  • Lo inicializa con Mutex() al entrar al constructor. Si la clase no tiene constructor de usuario, sintetiza uno por defecto que sólo hace esa inicialización.
  • Inserta this.__syncMutex.lock() al inicio y this.__syncMutex.unlock() al final de cada getX/setX.

Garantías: cada acceso individual al backing field es atómico respecto a otros threads. Una lectura nunca devuelve un estado parcial; una escritura es indivisible.

Importante: una expresión compuesta como this.value := this.value + 1 NO es atómica como un todo — se descompone en get + add + set, cada uno atómico individualmente, pero entre ellos otro thread puede colarse. Para operaciones atómicas compuestas usa un Mutex explícito alrededor del bloque.

Un return dentro del cuerpo custom de get/set sync libera correctamente el mutex: el emisor inyecta el unlock antes del salto al punto de salida (no hace falta try/finally explícito en BP). Lo mismo aplica a sync properties a nivel módulo.

Properties a nivel módulo

module Counter
  // Property pública del módulo. El estado vive en un global
  // privado, accesible desde fuera SÓLO a través del get/set.
  public property tick: integer
  public function bump()
    tick := tick + 1
  end bump
end Counter

module App
  import Counter
  function main(arg: string)
    Counter.bump()
    print Counter.tick           // lectura cross-module → 1
    Counter.tick := 42          // escritura cross-module
  end main
end App

BP permite declarar property también fuera de una clase, a nivel módulo. Es la forma canónica de exponer estado mutable a otros módulos: las var de módulo son privadas por diseño (no se exportan en la .bpi), pero una public property expone su getter/setter sintéticos, accesibles vía Modulo.nombre desde otro módulo que la importe.

Internamente el compilador emite:

  • Un global privado __prop_<nombre> que es el backing field.
  • Una función pública __prop_get_<nombre>() que devuelve el valor.
  • Una función pública __prop_set_<nombre>(__val) que lo escribe.
  • En el .bpi, una línea prop <nombre>:<tipo> public que viaja a los módulos importadores (sin exponer los accesores como funciones normales).

El identificador tick dentro del propio módulo resuelve al getter/setter (no al global directo), así que la encapsulación se respeta también dentro del módulo dueño. Si se necesita inicializar la property con un valor distinto de cero, hazlo en la función inicializadora del módulo (función con el mismo nombre que el módulo), igual que con var a nivel módulo.

sync también es válido en properties de módulo: 

module Stats
  public sync property contador: integer
end Stats

En este caso el módulo declara un único Mutex global (__prop_mutex) compartido por todas sus sync properties. Se inicializa al cargar el módulo (en __init) y el get/set envuelven su cuerpo con lock/unlock. Las garantías y limitaciones son las mismas que para sync en clases.

Limitaciones del MVP:

  • No se soportan properties owner a nivel módulo (sólo en clases).
  • No se soporta final a nivel módulo (el modificador sólo aplica a properties de clase).
  • El init expression de la property se ignora en compile-time; usa la función inicializadora del módulo para fijar el valor inicial.

9.6 static / final

static en una variable de clase: campo a nivel clase, no de instancia. final en clase: no extensible; en método: no overridable. (En BP el modificador es final antes de function o class.)

9.7 Fields owner

public class Caja
  public var owner contenido: Recurso
  public function Caja(r: Recurso)
    this.contenido := r
  end Caja
end Caja

Cuando una Caja se libera (via FREE_REF), su campo owner contenido se libera también, recursivamente.

10. Módulos

10.1 library opcional

library "com.example.app"
module Util
  ...
end Util

Si declaras library, el artefacto se nombra com.example.app.Util.mod y .bpi. Sin library, se llama Util.mod.

10.2 main(arg: string) y constructor de módulo

module MiApp
  // Initializer (auto): se ejecuta al cargar el módulo, tras los imports.
  function MiApp()
    print "[init] cargado"
  end MiApp

  function main(arg: string)
    print "ejecutando con arg='" + arg + "'"
  end main
end MiApp
  • function ModuleName() es el "constructor" del módulo. Se ejecuta tras inicializar todos los imports.
  • function main(arg: string) es el entry-point. Sólo se llama al main del módulo raíz; los mains de las dependencias quedan como utilidad para testing standalone.
  • Forma legacy: function Main() sin parámetros (sigue aceptada).

10.3 import / from

module App
  import Util                       // misma library
  import com.example.Util           // path cualificado
  import ext.tools.Helper from "plugins/helper.mod"
  ...
end App
  • El último segmento del path es el alias local (Util, Helper). Lo usas como Util.foo().
  • from "<path>" dice a la VM dónde encontrar el .mod en runtime. Compilable cuando el frontend localiza el .bp fuente derivando la ruta.

10.4 module interface

library "com.example"
module interface LogApi
  public function log(msg: string)
  public function level(): integer
  public const VERSION: integer := 1
end LogApi

Una interfaz es un contrato (Modula-2 DEFINITION MODULE): sólo firmas públicas, sin cuerpo de función. Compila a un .bpi con el flag interface true y no produce .mod.

Las interfaces pueden extender otra interfaz (herencia de contratos): 

module interface LogApiV2 extends com.example.LogApi
  public function logWithLevel(lvl: integer, msg: string)
  public function flush()
end LogApiV2

10.5 implements

module ConsoleLogger implements com.example.LogApi
  public function log(msg: string)
    print "[CON] " + msg
  end log
  public function level(): integer
    return 0
  end level
  public const VERSION: integer := 1
end ConsoleLogger

El analyzer verifica que el módulo expone TODAS las firmas del contrato (funciones con tipos idénticos, consts con mismo tipo, enums con mismos miembros y valores). El impl puede añadir libremente extras públicos no exigidos por el contrato. Errores típicos:

  • no implementa la función 'foo' requerida por 'I'
  • la función 'foo': retorno declarado 'float', se esperaba 'integer'
  • la función 'foo' requerida por 'I' debe ser public

10.6 Binding import I:Impl

module App
  import com.example.LogApi:ConsoleLogger
  // Equivalente con from:
  // import com.example.LogApi:ConsoleLogger from "plugins/logger.mod"

  function main(arg: string)
    LogApi.log("hola")         // tipa contra LogApi; CALL_EXT a ConsoleLogger.log
  end main
end App

Compile-time: el namespace LogApi se tipa contra la interfaz. Codegen: cada LogApi.x emite CALL_EXT al impl bindeado. Cambiar de impl = editar la línea del binding y recompilar.

Compatibilidad LSP: import LogApi:BufferedLogger es válido aunque BufferedLogger declare implements LogApiV2 y LogApiV2 extends LogApi. El driver recorre la cadena implements → extends.

10.7 Encapsulación clase ↔ módulo

Desde dentro de una clase, sólo se accede a símbolos públicos del módulo (functions, consts, vars, properties). Tipos (clases, enums, namespaces de imports) siempre visibles. 

module M
  function helperPriv(): integer     // privado
    return 99
  end helperPriv

  class C
    public function bad(): integer
      return helperPriv()             // ERROR: no es public
    end bad
  end C
end M

11. Excepciones

try
  riskyOp()
catch e: RuntimeError
  print "runtime: " + e.msg
catch e: MiError
  print "mi error: " + e.detalle
catch e
  print "desconocido"
finally
  print "limpieza"
endtry

throw RuntimeError("algo fallo")
  • try establece un handler en una pila de handlers de la VM.
  • catch e: Foo sólo atrapa instancias asignables a Foo. catch e (sin tipo) atrapa cualquier excepción.
  • finally se ejecuta siempre (con o sin excepción).
  • throw expr lanza el valor. Si es una clase, su class_ptr se usa para el match de catch tipado.
  • RuntimeError está siempre disponible (la sintetiza el emisor).
  • Una excepción no atrapada termina la VM mostrando un stack trace con el nombre simbólico de cada frame.

12. Owner semantics

BP soporta destrucción determinística de objetos en complemento al GC tradicional. La regla básica: un objeto puede tener un propietario; al liberarse el propietario, el objeto poseído también se libera (cascada).

12.1 var owner x: T

  • Al asignar x := y, el anterior valor de x se libera (FREE_REF) antes.
  • Al salir de scope, se libera el valor actual de x.
  • Asignar x a una variable no-owner copia la ref sin transferir propiedad (sigue siendo de x).
  • Asignar a otra variable owner transfiere: la fuente queda a null sin liberar.

12.2 Fields owner

public var owner f: T dentro de una clase. El descriptor de la clase guarda un bitmap de owners; al liberar la instancia, cada slot marcado se libera recursivamente. El opcode SET_FIELD_OWNER libera el valor anterior del campo antes de escribir el nuevo (asignación atómica owner-safe).

12.3 Cascada por arrays de refs

Si un campo owner contiene una referencia a un array TYPE_ARRAY_REF, al liberarse el contenedor se liberan también todos los slots no-null del array. Esta es la mecánica que sustenta OwnerList.

13.2 List

Lista dinámica de refs a objetos. Doubling al crecer. Backing items: integer[] es TYPE_ARRAY_REF, escaneado por el GC. add y set aceptan cualquier tipo de objeto (vía any); get devuelve any asignable a cualquier tipo de variable.

13.3 OwnerList

var owner ol: OwnerList := OwnerList()
ol.add(Recurso("A"))
ol.add(Recurso("B"))
ol.removeAndFree(0)               // libera 'A' determinísticamente
// al salir de scope: libera el array Y todos los items (cascada de FREE_REF)

Hereda de List. Su descriptor marca el field items como owner, así que al liberarse la lista (por var owner o por cascada) los items se liberan también. Métodos:

  • add, get, set, length, remove: heredados de List.
  • removeAndFree(idx): libera el objeto en items[idx] antes de desplazar el resto. remove(idx) no libera (se mantiene la API original).

13.7 Thread

class Worker extends Thread
  public var nombre: string
  public function Worker(nombre: string)
    super(2048)            // stackSize en bytes
    this.nombre := nombre
  end Worker
  public function run()
    var i: integer := 0
    while i < 5 do
      print this.nombre + " iter " + intToString(i)
      yield()
      i := i + 1
    endwh
  end run
end Worker

function main(arg: string)
  var w: Thread := Worker("A")
  w.start()
  w.join()
end main

Clase base de los threads BP. Sintetizada por el emisor; siempre disponible sin import. Para crear un thread subclasea Thread y sobrescribe run(). El constructor recibe el tamaño de stack (bytes; default interno 2 KB para workers, 16 KB para el main del programa).

MétodoDescripción
Thread(stackSize: integer) Constructor. La subclase debe llamarlo via super(N).
run() Cuerpo del thread. Override en la subclase. Cuando retorna, el thread termina. Slot 0 de la vtable (fijo por convención).
start() Arranca el thread: la VM aloca su región de stack, lo encola RUNNABLE y un worker Java lo recogerá.
join() Bloquea el thread llamador hasta que este thread termine. Si ya terminó es no-op.

Builtins relacionados (ver 13.1): yield() (cede CPU sin terminar) y sleep(ms) (cede + duerme N ms; cualquier otro thread RUNNABLE corre mientras). Para el azúcar parallel/case/default/endpar ver §7.6.

13.8 Mutex

var m: Mutex := Mutex()

// dentro del run() de cada Thread que comparta el mutex:
m.lock()                // bloquea si está tomado por otro
// sección crítica
m.unlock()              // libera y hand-off al primer waiter

Lock no reentrante para coordinar entre threads BP. Sintetizado por el emisor; siempre disponible sin import. Internamente la instancia guarda un id __mid que apunta a un mutex Java de la VM (JavaMutex bajo vmLock).

MétodoDescripción
Mutex() Constructor. Pide un id nuevo a la VM (__mutexCreate).
lock() Si libre, lo toma. Si lo tiene otro thread, marca al actual BLOCKED_MUTEX, lo añade a la cola de waiters y cede la CPU. Cuando reciba la propiedad, lock() regresa ya con el lock tomado.
unlock() Sólo legal desde el thread propietario; cualquier otro causa excepción. Si hay waiters, el primero recibe directamente la propiedad (hand-off, sin re-attempt) y pasa a RUNNABLE.

No reentrante: si el mismo thread vuelve a llamar lock() sobre un mutex que ya tiene, se lanza una excepción.

14. Concurrencia

La VM simula un microcontrolador de 2 cores: por defecto lanza N workers Java reales (default N=2; configurable con --workers=N, ver §15.2) que se reparten los threads BP encolados.

14.1 Modelo de ejecución

  • El thread BP main (tid 0) arranca primero: ejecuta el initializer del módulo y luego main(arg). Tiene stack grande (16 KB). Sólo él puede ejecutar la instrucción HALT que apaga la VM.
  • Cualquier subclase de Thread añade un thread BP nuevo al hacer start(). La VM le asigna stack (default 2 KB) y lo encola RUNNABLE. Un worker Java libre lo coge y ejecuta su run() hasta retorno (THREAD_EXIT) o yield()/sleep()/join()/lock.
  • Cada thread BP tiene su propio contexto (ThreadContext): PC, SP, BP, CS, región de stack, pila de handlers de excepción. El heap (incluyendo objetos owned por threads) es compartido.
  • Cuando todos los threads BP terminan, la VM se apaga. Si main hace HALT antes, los workers en vuelo se cortan en su próximo opcode.

14.2 Preempción y yield()

Un timer daemon (ScheduledExecutorService, quantum por defecto 10 ms) marca yieldRequested=true sobre todos los threads BP RUNNING. El bucle del intérprete lo observa entre opcodes (campo volatile) y abandona el bucle inner con señal YIELD; el worker re-encola el thread y coge el siguiente RUNNABLE.

En paralelo a la preempción, el código BP puede ceder CPU cooperativamente:

  • yield() — cede una vez; el thread sigue RUNNABLE y el scheduler lo elige cuando le toque.
  • sleep(ms) — marca BLOCKED_SLEEP hasta now() + ms. Si no hay otro RUNNABLE, el worker Java duerme hasta el próximo wake-up.
  • join() sobre otro thread — marca BLOCKED_JOIN hasta que el target termine.
  • Mutex.lock() sobre un mutex tomado — marca BLOCKED_MUTEX hasta hand-off.

14.3 Encapsulación de cuerpos de case

El cuerpo de cada case dentro de un parallel es el run() de una subclase anónima de Thread sintetizada por el compilador. Sigue la misma regla de encapsulación de objeto que un método de clase normal: no puede leer/escribir variables globales del módulo. Sí puede llamar funciones del módulo, instanciar clases, usar consts y llamar builtins.

Para coordinarse, los threads usan objetos compartidos (Mutex, List, clases con estado) en vez de globals mutables. El compilador rechaza la lectura directa con mensaje: 

error semántico: el cuerpo de un 'case' dentro de 'parallel' no puede
acceder a la variable global 'X'; usa una const, una función o un
objeto compartido

El bloque default NO sufre esta restricción: corre en el thread llamador y ve sus locales y globals.

14.4 GC y concurrencia

El GC mark-and-sweep es conservativo: escanea los stacks de todos los threads vivos y los data blocks de los módulos para identificar roots. Las operaciones de heap (heapAlloc, gc(), freeOwnedObject) están serializadas bajo vmLock — dos workers no pueden mutar el heap a la vez.

Safepoint stop-the-world: cuando un worker necesita GC (porque heapAlloc agota el heap), pone una bandera global stopTheWorld. Los demás workers la chequean al inicio de cada iteración del intérprete: sincronizan tc.{pc,sp,bp,cs} y abandonan runOnContext con señal YIELD. Vuelven a la WorkerLoop, ven la bandera y parquean en vmLock.wait(). El worker que pidió GC espera a que ninguno otro esté en estado RUNNING (los que están parqueados en vmLock.wait dentro de heapAlloc se cuentan como parqueados vía parkedInHeapAlloc), corre gcLocked, y luego limpia la bandera.

Además, cada tc mantiene una ancla de alloc (tc.allocAnchor): heapAlloc deja en ella el ref recién alocado antes de soltar vmLock. Si otro worker dispara GC entre el return de heapAlloc y el push posterior del caller, el escáner del GC ve la ancla como un root extra y no libera el objeto en tránsito.

Wraps de opcodes de alocación: NEW_OBJECT y las variantes de NEWARRAY mantienen vmLock desde la alocación hasta la publicación al stack BP (tc.sp sincronizado). Esto cierra la ventana en que otro worker podría sweepear el objeto recién creado antes de que su ref llegue al stack de quien lo creó.

Residual conocido (~5-10 % bajo contención muy alta): algunos builtins de allocación (allocVmString, allocVmRefArray, GROW_REF_ARRAY) todavía liberan vmLock entre la alocación y el push final. Cuando un patrón productor/consumidor con muchas allocaciones simultáneas se materializa, puede aparecer corrupción puntual. Síntomas: INVOKE_VIRTUAL sobre null, "Dirección 0", opcode 0xA6.

Cuando el residual se manifiesta, sólo muere el thread afectado — el resto del programa continúa porque BpThreadFault evita que un fallo individual tumbe la VM entera (ver el código de VirtualMachine.WorkerLoop).

  • synctest.bp (clase + sync property + 5000 iters × 2 threads): ~90 % pass-rate.
  • synclisttest.bp (SyncList + 1000 prods + 2 consumers): ~93 % pass-rate.
  • modpropsync.bp (sync property a nivel módulo): ~95 % pass-rate.

15.2 VM (bpgenvm)

java -jar bpgenvm.jar [opciones] <archivo.mod>
  • Por defecto carga y ejecuta el .mod raíz. La VM resuelve transitivamente los imports leyendo cada .mod referenciado.
  • --trace imprime PC/Opcode/CS/SP/tid por instrucción.
  • --workers=N número de worker threads Java en paralelo (default 2; equivale a "cores físicos simulados"). --workers=1 serializa todo: útil para reproducibilidad o depurar races. Ver §14.
  • -d hace dump del bytecode (disassembler integrado).
  • --config <archivo> carga config JSON (BpVM.cfg) desde la ruta indicada.
  • --no-config ignora cualquier BpVM.cfg auto-descubierto y arranca con defaults.

BpVM.cfg — configuración de la VM

{
  "memorySize": 524288,
  "stackBase":  262144,
  "stdlibDir":  "C:/lenguajes/pm/bpstdlib"
}

Fichero JSON con la configuración global de la VM. Todos los campos son opcionales; los que falten toman el valor por defecto:

  • memorySize (bytes totales del array memory; default 524288 = 512 KiB).
  • stackBase (offset donde empiezan los stacks; default 256 KiB — se valida que stackBase < memorySize y que quepa al menos MAIN_STACK_BYTES). Si pones memorySize pero no stackBase, se deriva a memorySize/2.
  • stdlibDir (path al directorio donde la VM busca los módulos stdlib — Json.mod hoy, más en el futuro — cuando un import no resuelve desde el cwd o desde el fromPath del importer).

El parser admite comentarios // hasta-fin-de-línea. No soporta arrays ni objetos anidados (no hace falta hoy).

Resolución:

  1. Si pasas --config <ruta>, se carga esa ruta. Si el fichero no existe o el JSON está mal formado, la VM aborta con mensaje claro.
  2. En caso contrario, la VM busca BpVM.cfg en:
    1. el directorio actual (cwd);
    2. el directorio donde está el .mod raíz.
    El primero que exista se usa.
  3. Si no hay nada, se usan los defaults.

El BpIde también respeta BpVM.cfg al pulsar F5 (Run) o F11 (Debug) — busca primero en el cwd y luego junto al .mod que está ejecutando.

.bpproject — fichero de proyecto de la VM

{
  "main":        "out/MyApp.mod",
  "modulePaths": ["out", "../shared/modules"]
}

Fichero JSON que describe qué .mod arrancar y dónde buscar los .mod de los imports. Se le pasa a la VM como argumento alternativo al .mod: 

bpgenvm MyApp.bpproject

La VM detecta el tipo por extensión (.mod vs. .bpproject). Campos:

  • main (string, obligatorio): ruta al .mod del módulo de arranque. Puede ser absoluta o relativa al directorio del .bpproject. La VM verifica que el fichero existe.
  • modulePaths (array de strings, opcional): directorios donde la VM busca los .mod de los imports, en orden de prioridad. Paths relativos se resuelven respecto al .bpproject.

Resolución completa de un import:

  1. Como esté indicado (cwd o absoluto).
  2. Cada modulePaths[i]/<basename> en orden.
  3. stdlibDir/<basename> (de BpVM.cfg).
  4. Si nada existe, FileNotFound con mensaje claro.

El parser admite comentarios //. Soporta arrays (a diferencia del BpVM.cfg, que es plano). Ver docs/BpProject.example para una plantilla anotada.

17. Pendientes / no soportado

  • Tipos compuestos en .bpi (clases, arrays no primitivos, properties) — cross-library hoy se limita a funciones, consts y enums con tipos primitivos.
  • Map<K,V> y otras estructuras en stdlib.
  • Versionado explícito de contratos (cubierto pragmáticamente con extends).
  • Genéricos / type parameters en clases del usuario.
  • TypeRef cualificado en BP (var c: Mod.Color).
  • Fichero de proyecto (.bpproj) para multi-impl coexistente en outDir con resolución a runtime.
  • JVM backend para cross-library / interfaces (sólo el backend mivm los soporta hoy).
  • Owners en caminos de excepción (algunos paths de throw saltan free).
  • Concurrencia — safepoint para GC stop-the-world: hoy gc() lee tc.sp de workers en vuelo. Bajo presión de heap con paralelismo intenso pueden no marcarse refs vivas a medio opcode. Ver §14.4.
  • Mutex reentrante / Semaphore / Channel en stdlib.
  • Closures sobre locales del encerrador en cuerpos de case del statement parallel (hoy sólo se ven stdlib + módulo no-globals — ver §14.3).
